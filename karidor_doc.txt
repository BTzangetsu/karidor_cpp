----------------------------------------------------------------
  CONVENTIONS GLOBALES
----------------------------------------------------------------

Indexation
  - Toutes les positions (cases, murs) sont 1-indexed.
  - (1, 1) est le coin supérieur gauche du board.
  - Les rangées vont de haut (1) vers le bas (n).
  - Les colonnes vont de gauche (1) vers la droite (n).

Coordonnées
  - On utilise toujours la convention (row, col) = (r, c).
  - row = rangée (axe vertical), col = colonne (axe horizontal).

Tailles de board selon le nombre de joueurs
  - 2 joueurs → board 9×9,  10 murs par joueur
  - 3 joueurs → board 11×11, 8 murs par joueur
  - 4 joueurs → board 13×13, 6 murs par joueur


----------------------------------------------------------------
  board.py
----------------------------------------------------------------

Rôle
  Contient toute la logique interne du jeu : représentation du
  plateau, déplacement des joueurs, placement des murs, et
  détection de victoire. C'est le "moteur" du jeu, indépendant
  de Flask et du front-end.

--- Classe Player ---

  Représente un joueur avec :
    - name  : identifiant du joueur
    - color : couleur de sa balle (utilisée par le front)
    - walls : nombre de murs qu'il lui reste à poser
    - position : tuple (r, c) 1-indexed, mis à jour à chaque mouvement

  Le nombre de murs est initialisé par Board.add_player() selon
  la taille du board — le joueur lui-même ne le fixe pas.

--- Classe Wall ---

  Représente un mur posé sur le board.
  Un mur couvre DEUX cases consécutives dans une tranchée.

  Convention de placement :
    La position (r, c) désigne le coin supérieur-gauche du mur.

    Mur VERTICAL Wall(r, c) :
      → tranchée à gauche de la colonne c
      → bloque le passage entre col (c-1) et col c
      → couvre les rangées r et r+1
      → bornes valides : 1 ≤ r ≤ n-1  et  1 ≤ c ≤ n+1
      → c=1 correspond au bord gauche du board (hors cases)
      → c=n+1 correspond au bord droit du board (hors cases)

    Mur HORIZONTAL Wall(r, c) :
      → tranchée au-dessus de la rangée r
      → bloque le passage entre rangée (r-1) et rangée r
      → couvre les colonnes c et c+1
      → bornes valides : 1 ≤ r ≤ n+1  et  1 ≤ c ≤ n-1
      → r=1 correspond au bord haut du board (hors cases)
      → r=n+1 correspond au bord bas du board (hors cases)

  Pourquoi autoriser les bords (c=1, c=n+1, r=1, r=n+1) ?
    Les joueurs doivent traverser le board jusqu'au bord opposé.
    Un joueur peut longer le bord — un mur sur la tranchée
    extérieure peut donc légitimement bloquer ce chemin.

--- Classe Board ---

  Attributs principaux :
    - size    : taille du board (9, 11 ou 13)
    - board   : grille (size+1) × (size+1) de booléens,
                True si la case est occupée. Index 0 ignoré.
    - players : liste des Player dans l'ordre d'ajout
    - walls   : liste des Wall posés
    - playerturn : index du joueur dont c'est le tour

  Méthode add_player(player)
    Place automatiquement le joueur sur son bord de départ :
      Joueur 0 → bord haut    : (1,    mid)
      Joueur 1 → bord bas     : (n,    mid)
      Joueur 2 → bord gauche  : (mid,  1)
      Joueur 3 → bord droit   : (mid,  n)
    mid = (size + 1) // 2 pour être exactement au centre.
    Initialise aussi son stock de murs selon WALLS_PER_PLAYER.

  Méthode make_move(r, c)
    Tente de déplacer le joueur courant vers (r, c).
    Valide si :
      1. Destination dans les limites (1..n)
      2. Case libre
      3. Mouvement orthogonal d'exactement 1 case (pas de diagonale)
      4. Aucun mur ne bloque le passage (via _is_blocked_by_wall)
    Si valide : libère l'ancienne case, occupe la nouvelle,
    passe au joueur suivant. Retourne True/False.

  Méthode _is_blocked_by_wall(from_pos, to_pos)  [privée]
    Pour un déplacement vertical (même colonne) :
      Cherche un mur HORIZONTAL dont la tranchée est exactement
      entre les deux rangées, et qui couvre la colonne du joueur.
    Pour un déplacement horizontal (même rangée) :
      Cherche un mur VERTICAL dont la tranchée est exactement
      entre les deux colonnes, et qui couvre la rangée du joueur.

  Méthode add_wall(wall)
    Tente de poser un mur. Validations dans l'ordre :
      1. Le joueur courant a encore des murs en stock
      2. Le mur est dans les limites du board
      3. Pas de mur identique déjà posé à cet endroit
      4. Pas de chevauchement partiel avec un mur adjacent
         de même orientation (deux murs ne peuvent pas partager
         une des deux cases qu'ils couvrent)
      5. BFS : le mur ne coupe pas entièrement le chemin
         d'au moins un joueur vers son bord de victoire
    Si valide : pose le mur, décrémente le stock, passe au suivant.

  Méthode _player_can_reach_goal(player_index)  [privée]
    BFS depuis la position actuelle du joueur.
    Explore les 4 directions orthogonales en ignorant les autres
    joueurs (qui peuvent se déplacer), mais en respectant les murs.
    Retourne True si le bord de victoire est atteignable.

  Méthode _goal_rows_cols(player_index)  [privée]
    Retourne la condition de victoire du joueur :
      Joueur 0 → atteindre rangée n   (traverser vers le bas)
      Joueur 1 → atteindre rangée 1   (traverser vers le haut)
      Joueur 2 → atteindre colonne n  (traverser vers la droite)
      Joueur 3 → atteindre colonne 1  (traverser vers la gauche)

  Méthode check_winner()
    Parcourt tous les joueurs et retourne l'index du premier
    qui a atteint son bord de victoire, ou None si personne.
    À appeler après chaque make_move().


----------------------------------------------------------------
  game.py
----------------------------------------------------------------

Role
  Game is the manager of a single game session. It wraps a Board
  instance and adds everything above the raw board logic:
  lobby state, game lifecycle, and a clean interface that the
  socket event handlers will call directly.

  The game loop does NOT live here. In a WebSocket app the
  "loop" is the network itself:
    client clicks → socket event → events.py → game.py → broadcast

Attributes
  - num_players  : number of players expected before the game starts
  - board        : the Board instance (holds all grid logic)
  - started      : False while waiting for players in the lobby,
                   True once all expected players have joined
  - finished     : True once a winner has been found
  - winner       : index of the winning player in board.players,
                   None while the game is ongoing
  - game_code    : 4-digit string used by other players to join
                   (e.g. "4782"), generated randomly at creation

Methods

  _get_board_size() → int
    Maps num_players to the correct board size (9 / 11 / 13).

  _generate_code() → str
    Returns a random 4-digit string between "1000" and "9999".

  join(name, color) → bool
    Called when a player enters the lobby code and joins.
    - Rejects if the game has already started.
    - Creates a Player object and calls board.add_player().
    - Once len(board.players) == num_players, sets started = True.
    - Returns True on success, False on failure.

  play_move(r, c) → bool
    Called by events.py when the current player moves their ball.
    - Guards: must be started and not finished.
    - Calls board.make_move(r, c) with two separate ints (not a tuple).
    - If the move is valid, calls check_winner() to detect end of game.
    - Returns True if the move was accepted, False otherwise.

  play_wall(r, c, vertical) → bool
    Called by events.py when the current player places a wall.
    - Guards: must be started and not finished.
    - Builds a Wall object and calls board.add_wall().
    - Returns True if the wall was placed, False otherwise.

  get_state() → dict
    Returns the full game state as a dict, broadcast to all clients
    via SocketIO after every valid action. Contains:
      - game_code    : the lobby code
      - started      : bool
      - finished     : bool
      - winner       : int or None
      - current_turn : index of the player whose turn it is
      - players      : list of {name, color, position, walls_left}
      - walls        : list of {position, vertical}


----------------------------------------------------------------
  rooms.py
----------------------------------------------------------------

Role
  RoomManager is the global registry of all active game sessions.
  While game.py manages the logic of a single game, RoomManager
  manages ALL games simultaneously and links socket IDs to players.

  A single RoomManager instance is created at app startup and
  shared across all socket event handlers.

Attributes
  - games       : dict[game_code, Game]
                  All active Game instances, keyed by their 4-digit code.

  - sid_to_game : dict[sid, (game_code, player_index)]
                  Maps each connected socket ID to the game it belongs to
                  and the index of the player it represents.
                  This is needed because socket events only provide the sid —
                  we must look up which game and which player it corresponds to.

Methods

  create_game(num_players) → Game
    Instantiates a new Game, stores it in self.games under its
    game_code, and returns it. Called when a player creates a lobby.

  get_game(game_code) → Game | None
    Returns the Game for the given code, or None if not found.
    Used as the first step before any game action.

  join_game(game_code, name, color) → bool
    Fetches the game via get_game(), then calls game.join().
    Returns False if the game doesn't exist or join() fails.

  register_player(sid, game_code, player_index) → None
    Stores sid → (game_code, player_index) in sid_to_game.
    Must be called immediately after a successful join_game()
    so that future events from this client can be routed correctly.

  get_game_by_player(sid) → tuple[Game, int] | None
    The most important lookup method. Given a socket ID, returns
    (Game, player_index) or None if the sid is unknown.
    Used at the start of every game event handler.

  unregister_player(sid) → None
    Removes a sid from sid_to_game.
    Called on SocketIO disconnect to clean up stale references.

  remove_game(game_code) → None
    Deletes the Game from self.games and removes all sid_to_game
    entries that pointed to it, avoiding dangling references.
    Called when a game ends or is abandoned.


----------------------------------------------------------------
  __init__.py  (app/)
----------------------------------------------------------------

Role
  Application factory. Centralizes app creation so the same
  factory can be reused for testing or multiple configs.
  Keeps the global SocketIO instance importable by events.py.

  create_app() :
    - Creates the Flask app with correct static/template folders
    - Calls socketio.init_app(app) to bind SocketIO
    - Registers the HTTP blueprint (routes.py)
    - Calls register_events(socketio) to bind all socket handlers
    - Returns the configured app

  socketio is instantiated at module level so events.py can
  import it without creating a circular import.


----------------------------------------------------------------
  routes.py  (app/)
----------------------------------------------------------------

Role
  Defines the two HTTP routes served by Flask.
  All real-time game logic goes through SocketIO — HTTP only
  serves the HTML pages.

  GET /       → renders index.html  (lobby: create or join)
  GET /game   → renders game.html   (board UI + SocketIO client)


----------------------------------------------------------------
  run.py
----------------------------------------------------------------

Role
  Entry point. Imports the factory and the SocketIO instance,
  creates the app, and starts the server.

  socketio.run() is used instead of app.run() so that the
  WebSocket transport is handled correctly by eventlet.

  use_reloader=False prevents the Flask reloader from spawning
  a second process that would conflict with SocketIO's event loop.

  host='0.0.0.0' makes the server reachable from other devices
  on the network (and from inside Docker).
